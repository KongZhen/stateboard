产品总览

从散落上下文，到可行动信号。一张持续更新的项目状态板。

Stateboard 把 AI 对话、项目文件、项目说明、待办和验证结果整理成持续更新的项目状态板。 它在工作发生时保留证据，让下一步一直可见，并且只在边界清楚时同步到目标工具。

Stateboard Project · GitHub Pages · alpha

  • 不是 todo 清单。它记录发生了什么、为什么重要、下一步该动什么。
  • 先证据。每一行都应该能追溯到文档、说明、输出或验证结果。
  • 同步有边界。只有目标、层级和时间清楚后，才写入外部系统。

“目标不是更多 tracking，而是更少重复解释。”

01

不是 todo assistant

Stateboard 不是待办清单或任务管理器。它是对工作上下文的状态反映：发生了什么、为什么重要、下一步是什么。

  • 保留决策、输出、链接和学习。
  • 浮现真正应该执行的下一步。
  • 让历史保持可见和可搜索。
02

证据工作流

证据是基础。每次更新都应该能追溯到 note、log、output 或可打开的链接。

  • 边工作边捕捉上下文。
  • 把 artifact 链接回仓库。
  • 在同步边界重新检查。
03

快速开始

先不安装服务，也能在仓库或当前线程里得到一张状态板。

  1. 复制 prompt。
  2. 在 AI assistant 中运行。
  3. 提交生成的 board，或同步到已确认的目标。
04

集成层级

集成按深度和可靠性描述，不写模糊的平台广度承诺。

  • 当前：prompt 和 Skill。
  • 已配置：有真实 connector 时才写入。
  • Fallback：Markdown 或导出行。
快速开始：复制 prompt 一个 prompt，任何 assistant。
prompts/stateboard-sync-prompt.zh.md 
请帮我把下面这些项目/对话内容整理成一个项目状态板，并同步到我指定的位置。

我想整理的内容：
【例如：我近 60 天能提供或你能读取的 Codex / ChatGPT / Claude / DeepSeek / WorkBuddy 对话；当前这个项目；某个项目文件夹；我已有的待办；几段粘贴的项目聊天记录】

同步到：
【例如：滴答清单 / Notion / Google Calendar / 飞书 / 钉钉 / 企业微信 / 只输出 Markdown】

目标位置：
【例如：滴答清单里的“项目总控”清单；Notion 的某个页面/数据库；如果没有就写“请新建一个叫 XXX 的清单”；只输出 Markdown 就写“直接发给我”】

自动更新：
【不需要 / 每天 HH:MM / 每周X HH:MM。没有明确时间就写“不需要”】

请按这些规则执行：
- 只整理你能读取、搜索到，或我已经提供给你的内容；如果你看不到某些对话或项目资料，请告诉我还需要补充什么。
- 先根据证据判断项目状态，不要把聊天记录直接拆成一堆任务。
- 默认每个真实项目只创建或更新 1 条项目状态任务，里面放当前进展、风险/阻塞、下一步和证据。
- 每条任务只保留 3-5 个真正有用的下一步。
- 写入前先搜索已有任务；如果已有同一个项目的状态任务，请更新它，不要重复创建。
- 如果目标工具当前不能写入，就先输出可复制的 Markdown 版本，并说明卡在哪里。
- 如果我要自动更新，但没有写清楚具体频率和时间，请先问我，不要默认任何时间；如果当前环境不能创建自动更新，就给我一份可复制的设置说明，不要假装已经创建。
- 不要写入 token、webhook、license key、私有凭据或私有目标系统标识。
- 默认用我这次请求的语言回复，并用同一语言写入目标系统；只有我明确要求时才输出双语。
- 完成后告诉我：创建了什么、更新了什么、如何验证、还有什么需要我确认。

集成层级

深度优先于广度。

Stateboard 不会在没有层级说明时说“支持所有工具”。任务同步、日历同步、通知投递和 Markdown fallback 是不同承诺。

  • 当前入口是 prompt 和 Skill。
  • 真实写入必须先发现 connector 并完成验证。
  • 自动化必须等待用户确认频率和具体时间。
层级 重点 例子 能力 可用性
当前 本地优先 Prompt、Skill、Markdown 从可读取上下文生成项目状态板和运行报告。 现在
Tier 1 直接 connector 已配置的滴答清单 / Notion / Calendar 在当前运行环境中写入、搜索、更新并验证。 取决于运行环境
Tier 2 官方 API 路径 飞书 / Lark、钉钉、自定义 MCP 需要应用配置、权限、token 和 ID 映射。 规划中
Tier 3/4 通知或导出 企业微信 digest、CSV-like rows、复制导入 适合投递或 fallback，不是完整任务数据库。 有边界
