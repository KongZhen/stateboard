Product overview

From scattered context to signal. A board that stays current.

Stateboard turns AI conversations, project files, notes, todos, and verification results into a living project-status board. It captures evidence as work happens, keeps next actions visible, and syncs knowledge at the right boundary.

By Stateboard Project · GitHub Pages · alpha

  • Not a todo list. It records what happened, why it matters, and what should move next.
  • Evidence first. Rows link to docs, notes, outputs, and verification results.
  • Sync boundary. Share only when the destination, tier, and timing are clear.

“The goal is not more tracking. It is less re-explaining.”

01

Not a todo assistant

Stateboard is not a todo list or task manager. It is a reflection of your work context, summarized into what happened, why it matters, and what is next.

  • Captures decisions, outputs, links, and learnings.
  • Surfaces the right next actions.
  • Keeps history visible and searchable.
02

Evidence workflow

Evidence is the foundation. Every update should be backed by a note, log, output, or link you can open.

  • Capture context as you go.
  • Link artifacts in the repo.
  • Review at the sync boundary.
03

Quickstart

Get a board in your repo or thread without installing a service first.

  1. Copy the prompt.
  2. Run it in your AI assistant.
  3. Commit the generated board or sync to a confirmed destination.
04

Integration tiers

Integrations are described by depth and reliability instead of vague platform breadth.

  • Current: prompt and Skill.
  • Configured: real connectors when available.
  • Fallback: Markdown or export rows.
Quickstart: copy the prompt One prompt. Any assistant.
prompts/stateboard-sync-prompt.md 
Please organize the following projects or conversation context into a project-status board, then sync it to the destination I specify.

What I want to organize:
[For example: the last 60 days of Codex / ChatGPT / Claude / DeepSeek / WorkBuddy conversations that I can provide or you can read; the current project; a project folder; my existing todos; several pasted project chats]

Sync to:
[For example: Dida/TickTick, Notion, Google Calendar, Feishu/Lark, DingTalk, WeCom, or Markdown only]

Target location:
[For example: the "Project Control" list in Dida/TickTick; a Notion page/database; "create a new list called XXX"; if Markdown only, write "reply directly here"]

Automatic refresh:
[Not needed / daily HH:MM / weekly weekday HH:MM. If there is no clear time, write "not needed"]

Rules:
- Only use content you can read, search, or that I have provided. If some conversations or project materials are unavailable, tell me what is missing.
- Judge project state from evidence first. Do not split chat history directly into many tasks.
- By default, create or update only one project-status task per real project. Include current progress, risks/blockers, next steps, and evidence.
- Keep only 3-5 useful next steps per task.
- Search for existing tasks before writing. If a status task for the same project already exists, update it instead of creating a duplicate.
- If the target tool cannot be written to from the current environment, output a copy-ready Markdown version and explain the blocker.
- If I ask for automatic refresh but did not provide both frequency and exact time, ask me first. Do not assume any default time. If the current environment cannot create automation, give me copy-ready setup instructions instead of claiming it was created.
- Do not write tokens, webhook secrets, license keys, private credentials, or private destination identifiers.
- Respond and write destination content in the language of my request unless I explicitly ask for bilingual output.
- When done, tell me what was created, what was updated, how it was verified, and what still needs my confirmation.

Integration tiers

Depth over breadth.

Stateboard never says “supports everything” without a support tier. Todo sync, calendar sync, notification delivery, and Markdown fallback are different promises.

  • Current entry points are prompt and Skill.
  • Real writes require connector discovery and verification.
  • Automation waits for a confirmed frequency and exact time.
Tier Focus Examples Capability Availability
Current Local-first Prompt, Skill, Markdown Create a project board and run report from readable context. Now
Tier 1 Direct connector Dida/TickTick, Notion, Calendar when configured Write, search, update, and verify inside the current runtime. Runtime-dependent
Tier 2 Official API path Feishu/Lark, DingTalk, custom MCP Requires app setup, permissions, tokens, and ID mapping. Planned
Tier 3/4 Notification or export WeCom digest, CSV-like rows, copy/import Useful for delivery or fallback, not a full task database. Bounded
