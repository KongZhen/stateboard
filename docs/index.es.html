Resumen del producto

Del contexto disperso a señales accionables. Un tablero que se mantiene al día.

Stateboard convierte conversaciones con IA, archivos de proyecto, notas, tareas y resultados de verificación en un tablero vivo de estado de proyectos. Conserva evidencia mientras el trabajo ocurre, mantiene visibles las próximas acciones y sincroniza conocimiento solo cuando el límite está claro.

Stateboard Proyecto · GitHub Pages · alpha

  • No es una lista de tareas. Registra qué pasó, por qué importa y qué debería avanzar.
  • La evidencia va primero. Cada fila debe poder rastrearse hasta documentos, notas, salidas o resultados verificados.
  • La sincronización tiene límites. Solo escribe en sistemas externos cuando destino, nivel y horario están claros.

“El objetivo no es hacer más tracking, sino explicar menos veces lo mismo.”

01

No es un todo assistant

Stateboard no es una lista de tareas ni un gestor de tareas. Es una capa de estado que resume qué ocurrió, por qué importa y qué sigue.

  • Conserva decisiones, salidas, enlaces y aprendizajes.
  • Hace visibles las próximas acciones correctas.
  • Mantiene el historial visible y buscable.
02

Workflow de evidencia

La evidencia es la base. Cada actualización debe estar respaldada por una nota, log, salida o enlace que puedas abrir.

  • Captura contexto mientras trabajas.
  • Conecta artifacts al repositorio.
  • Revisa en el límite de sincronización.
03

Inicio rápido

Obtén un tablero en tu repo o hilo sin instalar un servicio primero.

  1. Copia el prompt.
  2. Ejecútalo en tu assistant de IA.
  3. Guarda el board generado o sincroniza con un destino confirmado.
04

Niveles de integración

Las integraciones se describen por profundidad y confiabilidad, no por amplitud vaga de plataformas.

  • Actual: prompt y skill de Codex.
  • Configurado: escribir solo cuando exista un connector real.
  • Fallback: Markdown o filas exportables.
Inicio rápido: copia el prompt Un prompt. Cualquier assistant.
prompts/stateboard-sync-prompt.md copy-ready 
Organiza los siguientes proyectos o contexto de conversación en un tablero de estado de proyectos y sincronízalo con el destino que especifique.

Qué quiero organizar:
[Por ejemplo: conversaciones de los últimos 60 días de Codex / ChatGPT / Claude / DeepSeek / WorkBuddy que pueda proporcionar o que puedas leer; el proyecto actual; una carpeta de proyecto; mis tareas existentes; varias conversaciones pegadas]

Sincronizar con:
[Por ejemplo: Dida/TickTick, Notion, Google Calendar, Feishu/Lark, DingTalk, WeCom o solo Markdown]

Ubicación de destino:
[Por ejemplo: la lista “Project Control” en Dida/TickTick; una página/base de datos de Notion; “crear una lista nueva llamada XXX”; si es solo Markdown, escribe “responde aquí”]

Actualización automática:
[No hace falta / diario HH:MM / semanal día HH:MM. Si no hay una hora clara, escribe “no hace falta”]

Reglas:
- Usa solo contenido que puedas leer, buscar o que yo haya proporcionado.
- Juzga el estado del proyecto desde evidencia primero. No conviertas el historial de chat en muchas tareas.
- Por defecto, crea o actualiza solo una tarea de estado por proyecto real.
- Busca tareas existentes antes de escribir. Si el mismo proyecto ya existe, actualízalo.
- Si la actualización automática no tiene frecuencia y hora exacta, pregunta primero. No asumas una hora predeterminada.
- No escribas tokens, secretos de webhook, license keys, credenciales privadas ni identificadores privados de destino.
- Al terminar, dime qué se creó, qué se actualizó, qué se verificó y qué necesita confirmación.

Niveles de integración

Profundidad antes que amplitud.

Stateboard no dice “lo soporta todo” sin un nivel de soporte. Sync de tareas, calendario, notificaciones y Markdown fallback son promesas distintas.

  • Las entradas actuales son prompt y skill de Codex.
  • Las escrituras reales requieren descubrir connector y verificar.
  • La automatización espera frecuencia y hora exacta confirmadas.
